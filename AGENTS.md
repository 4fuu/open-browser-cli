# browser-cli AGENTS

## 项目定位

`browser-cli` 是一个面向命令行和 AI 的浏览器会话操作工具。

它关注的是：

- 显式会话管理
- 结构化页面视图
- 基于元素 ID 的交互
- 可复用的自动化规则与插件扩展

它不做：

- 通用网页抓取
- 原始 HTML 搬运
- 无状态浏览器自动化
- 视觉层截图/识别

代码是实现细节的唯一准源。
`AGENTS.md` 只保留项目总览、关键不变量和协作规则，不重复代码里已经能直接读出的内容。

---

## 架构总览

整体分三层：

1. Rust CLI
   负责命令入口、页面结构化、文本检索、块分页、插件执行和输出格式化。

2. Relay
   负责本地桥接与快照缓存，本身尽量不做页面语义判断。

3. 浏览器扩展
   负责 DOM 采集、真实点击/输入/等待，并把原始快照回传给 Relay。

设计原则：

- 浏览器端尽量薄，只采原始事实并执行交互
- 主要语义判断放在 Rust 侧
- Relay 保持轻量，不成为业务逻辑中心

## 代码地图

- `src/cli/`
  用户命令、参数分发、输出。

- `src/page/`
  页面结构化核心。
  `structure.rs` 负责 `RawSnapshot -> PageData`；
  `xml.rs` 负责 XML 渲染。

- `src/relay/`
  本地 TCP / Native Messaging 桥接与快照缓存。

- `src/protocol/`
  Rust 与扩展共享的数据结构定义。

- `src/plugin/`
  插件加载与执行。

- `extension/src/content/`
  DOM 采集与交互执行。

- `extension/src/background/`
  与 Native Messaging、tab、session 的桥接。

## 核心说明

### 1. 会话是显式且持久的

会话由用户显式打开和关闭。
只要会话存活，页面状态、登录态、输入态、历史状态都应保留。

### 2. 页面输出是“结构化视图”，不是 DOM 镜像

输出目标是让人和 AI 读得懂、用得上，而不是把 DOM 原样复刻出来。

优先级：

- 保留真实可操作元素
- 保留必要文本与分组语义
- 尽量减少展示型噪声

### 3. 元素 ID 只对当前页面快照有效

`eN` / `tN` / `bN` 都是当前页面视图内的短期编号，不是稳定 DOM 标识。

不要设计任何依赖“旧 ID 在新页面仍然有效”的行为。

### 4. 浏览器端只提供事实，不做重语义推断

Content Script 负责：

- 采集标签、文本、属性、坐标等原始信息
- 执行 click / type / wait

它不应该承担复杂的页面结构化、文本压缩、业务字段命名等职责。

### 5. 核心层避免站点特判

通用规则可以增强，但不要把网站私有结构硬编码进核心逻辑。

优先顺序：

1. 原生语义
2. 无障碍 / 可聚焦语义
3. 保守的弱交互信号
4. 插件扩展

如果某种语义只能通过站点专有结构推断，优先放到插件，不放核心。

### 6. 语义保留优先于“绝对扁平”

压缩输出是为了降噪，不是为了丢语义。

例如：

- 单叶子包装层可以压平
- 记录边界、列表项边界、表格边界应尽量保留
- 无意义容器可以隐藏，但不要把同一条记录的字段打散到无法理解

### 7. `click` 只对当前可解析、可定位的元素负责

`click` 是真实交互，不是模糊导航捷径。

- 能定位到当前元素，就执行点击
- 对链接的新会话打开是显式分支
- 对虚拟滚动、事件委托、站点私有交互等场景，不要假装“总能点到”

---

## 交互识别策略

核心层应优先识别这些信号：

- 原生交互标签，如链接、按钮、表单控件
- 明确的 ARIA / role 语义
- 可聚焦且带状态的弱交互信号
- 显式 `onclick` 信号

但要避免把纯展示节点大面积误判为可交互。

如果页面交互性来自：

- 事件委托
- 私有前端框架模式
- 站点专有 class 组合

这类能力默认不应靠核心强猜，后续优先用插件扩展。

## 插件扩展计划（待实现）

插件用于承接不适合进入核心的增强，尤其是：

- 站点级字段命名
  例如把一组通用文本解释成“排名 / 作者 / 播放量”

- 站点级交互补强
  例如某些站点的特殊 tab、卡片、虚拟列表行为

- 自动化规则
  例如弹窗关闭、常见按钮点击、页面初始化动作

原则：

- 核心负责通用结构
- 插件负责站点知识

## 协作规则

### 协议或快照字段变更

如果修改了 `RawNode` / `RawSnapshot` / 扩展采集属性：

- Rust 与 TypeScript 两侧定义必须同步
- 相关结构化逻辑必须同步
- 补最小必要测试

### 页面结构或 XML 变更

如果修改了结构化规则或 XML 输出：

- 优先加/改 `src/page/structure.rs` 和 `src/page/xml.rs` 的单测
- 先验证语义是否更清晰，再考虑是否更短

### 扩展侧变更

如果改了 Content Script：

- 运行 `npm run typecheck`
- 运行 `npm run build`
- 需要真实页面验证时，记得 reload 浏览器扩展并重新抓 `--fresh` 快照

### Rust 侧变更

完成后至少运行：

- `cargo test`
